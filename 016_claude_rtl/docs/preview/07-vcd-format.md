# 07 · VCD 포맷 · 생성기 설계

## 원칙: RTL-driven

VCD는 RTL의 dump 시스템 태스크가 호출될 때만 생성된다. `$dumpfile`과 `$dumpvars`를 호출하지 않은 시뮬레이션은 VCD를 일절 만들지 않는다 — no-op. 시뮬레이터가 임의로 dump를 시작하는 일은 없다.

이 원칙이 생성기 설계의 출발점이다. `VcdWriter`는 dump 태스크 핸들러에 의해서만 활성화되는 수동적 컴포넌트다.

---

## 인식·지원할 dump 시스템 태스크

아래 6가지를 Phase 1에서 전부 구현한다. `$dumpfile` + `$dumpvars`만으로 기본 파형 확인이 가능하고, 나머지 4개는 장시간 시뮬레이션과 디버그 시나리오에 필요하다.

| 태스크 | 구문 예시 | 동작 |
|--------|-----------|------|
| `$dumpfile` | `$dumpfile("sim.vcd")` | VCD 출력 파일 경로 지정 |
| `$dumpvars` | `$dumpvars` / `$dumpvars(0, tb)` / `$dumpvars(2, tb.dut)` | 덤프 대상 지정. 첫 인수=깊이(0=무한), 이후=scope. 인수 없으면 전체 |
| `$dumpon` | `$dumpon` | `$dumpoff`로 정지된 덤프 재개, 현재 값 기록 |
| `$dumpoff` | `$dumpoff` | 덤프 일시 정지. 모든 추적 변수에 `x`/`bx` 기록 후 변화 중단 |
| `$dumpall` | `$dumpall` | 현재 시각의 모든 추적 변수 현재값을 즉시 강제 기록 (체크포인트) |
| `$dumpflush` | `$dumpflush` | 내부 VCD 버퍼를 파일로 즉시 플러시 |

추가로 `$dumplimit(bytes)` 도 지원한다. 파일 크기가 지정 바이트에 도달하면 덤프를 중단하고 `$comment Dump limit reached $end`를 삽입한다.

## 파일 구조 (golden 명세, IEEE 1364 §18)

### 헤더

VCD 파일은 `$keyword ... $end` 블록으로 구성된 헤더로 시작한다. 모든 키워드는 `$`로 시작하고 반드시 `$end`로 닫힌다.

```vcd
$date
   2026-05-28 09:00:00
$end
$version
   vitamin-sim 0.1.0
$end
$comment
   Generated by vitamin RTL simulator
$end
$timescale 1ns $end
$scope module tb $end
$scope module dut $end
$var wire 8 ! data $end
$var wire 1 " data_valid $end
$var reg  4 # state $end
$var wire 1 $ clk $end
$var wire 1 % rst_n $end
$upscope $end
$upscope $end
$enddefinitions $end
```

**토큰별 규칙:**

- `$date`, `$version`, `$comment` — 자유 형식 텍스트, 생략 가능
- `$timescale` — 허용 magnitude: `1`/`10`/`100`, 허용 단위: `s`/`ms`/`us`/`ns`/`ps`/`fs`. 예: `1ns`, `10ps`, `100fs`. 공백 유무 무관 (`1ns`와 `1 ns` 모두 허용).
- `$scope <type> <name> $end` — 계층 열기. type은 아래 표 참조
- `$upscope $end` — 상위 계층으로 복귀
- `$var <type> <width> <id_code> <reference> $end` — 변수 선언
- `$enddefinitions $end` — 헤더 종료. 이후부터 simulation data

**스코프 타입 (IEEE 1364 §18):**

| 타입 | 설명 |
|------|------|
| `module` | 모듈 인스턴스 |
| `task` | Verilog task |
| `function` | Verilog function |
| `begin` | named begin 블록 |
| `fork` | fork 블록 |

**변수 타입 18종 (IEEE 1364 §18):**

```
event  integer  parameter  real  realtime  reg
supply0  supply1  time  tri  triand  trior
trireg  tri0  tri1  wand  wire  wor
```

vitamin Phase 1에서 실제 사용하는 주요 타입: `wire`, `reg`, `integer`, `real`, `parameter`. 나머지는 선언 시 VCD에 그대로 기록하지만 동작은 `wire`/`reg`와 동일하게 처리한다.

### 초기 덤프 + 값 변화

`$enddefinitions $end` 이후 `$dumpvars` 블록으로 초기 상태를 기록한다. 아직 초기화되지 않은 변수는 `x`(unknown)를, 이미 값이 결정된 변수는 실제 값을 기록한다.

```vcd
$dumpvars
bxxxxxxxx !
x"
x#
0$
1%
$end
#0
b00000000 !
1"
b0000 #
0$
0%
#10
1$
#20
0$
1%
#30
b00000001 !
1"
b0001 #
```

주의: VCD 파일은 반드시 **마지막 레코드 뒤에 newline**으로 종료해야 한다. GTKWave는 파일 끝 newline 누락 시 버퍼 오버런으로 "Unknown VCD identifier" 오류를 발생시킨다 (GTKWave issue #336).

### 식별자 코드

`$var` 선언에서 각 변수에 고유한 식별자 코드를 할당한다. 이후 값 변화 레코드에서 변수 이름 대신 이 짧은 코드를 사용해 파일 크기를 줄인다.

- **문자 범위**: printable ASCII 33(`!`)~126(`~`), 공백(32) 제외
- **길이**: 1~4문자 (구현에 따라 더 길 수 있으나 관례상 1~2자)
- **인코딩 알고리즘** (Tcl VCD 구현 기반 — 다중 카운터 방식):

```
초기화: c0=32, c1=32, c2=32, c3=32

변수 추가 시:
  c0 += 1
  if c0 >= 127: c1 += 1; c0 = 33
  if c1 >= 127: c2 += 1; c1 = 33
  if c2 >= 127: c3 += 1; c2 = 33

ID 문자열 = [c3 if c3>32] + [c2 if c2>32] + [c1 if c1>32] + char(c0)
```

결과: 첫 변수 `!`, 두 번째 `"`, ..., 94번째 `~`, 95번째 `"!`, ..., 최대 94^4 ≈ 78M 변수.

주의: 이 알고리즘은 Tcl VCD 패키지에서 확인한 단일 구현이다. IEEE 1364 §18은 인코딩 알고리즘 자체를 규정하지 않고 "printable ASCII 33-126 범위의 1개 이상 문자"만을 요건으로 한다. 결과 범위는 표준 적합이고 rust-vcd의 `IdCode::next()` 방식과도 일관성이 있다.

### 값 표기

세 가지 값 표기 형식을 엄격히 구분한다:

**스칼라 (1비트):**
```
<value><id_code>
```
- `value`: `0`, `1`, `x`, `z` 중 하나
- `value`와 `id_code` 사이에 **공백 없음** — 이 규칙은 GTKWave 호환에 필수
- 예: `0$`, `1%`, `x!`, `z"`

**벡터 (다비트):**
```
b<binary_value> <id_code>
```
또는 대문자 `B`도 허용
- `b`와 이진수 사이에 공백 없음
- 이진수와 `id_code` 사이에 **공백 필수**
- 예: `b10000001 !`, `bxxxxxxxx !`, `b0 #`

**실수 (real 타입):**
```
r<value> <id_code>
```
또는 대문자 `R`도 허용
- 값은 `printf %.16g` 형식
- 값과 `id_code` 사이에 **공백 필수**
- 예: `r3.14159265358979 !`

### `$comment` 블록

```vcd
$comment
   임의 텍스트. 여러 줄 가능. $end 가 나타날 때까지 계속.
$end
```

`$comment`는 헤더뿐 아니라 simulation data 섹션 어디서든 삽입 가능하다. `$dumplimit` 도달 시, `$dumpoff`/`$dumpon` 이벤트 기록 등에 활용한다.

## 생성기 설계 (vcd-writer 크레이트)

### 인터페이스

vitamin은 `vcd-writer`라는 내부 크레이트를 구현한다. rust-vcd crate의 `Writer<W>` 구조에서 영감을 받되, vitamin의 시뮬레이션 루프와 `hdl-builtins` dump 핸들러에 맞게 설계한다.

```rust
use std::io::Write;
use std::collections::HashMap;

/// VCD 식별자 코드. printable ASCII 33-126 기반 다문자 인코딩.
#[derive(Clone, Copy, Debug, PartialEq, Eq, Hash)]
pub struct IdCode(u32);

impl IdCode {
    pub const FIRST: Self = IdCode(0);

    /// 다음 ID 생성 (카운터 증가)
    pub fn next(self) -> Self {
        IdCode(self.0 + 1)
    }
}

impl std::fmt::Display for IdCode {
    /// ASCII 33-126 범위 다문자 인코딩으로 출력
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        let mut n = self.0 as u64;
        let mut chars = Vec::new();
        loop {
            chars.push(char::from_u32(33 + (n % 94) as u32).unwrap());
            n /= 94;
            if n == 0 { break; }
            n -= 1;
        }
        for c in chars.iter().rev() {
            write!(f, "{}", c)?;
        }
        Ok(())
    }
}

/// VCD 변수 타입 (IEEE 1364 §18)
#[derive(Clone, Copy, Debug)]
pub enum VarType {
    Wire, Reg, Integer, Real, Realtime,
    Event, Parameter, Time,
    Supply0, Supply1, Tri, Triand, Trior, Trireg, Tri0, Tri1,
    Wand, Wor,
}

/// VCD 스코프 타입 (IEEE 1364 §18)
#[derive(Clone, Copy, Debug)]
pub enum ScopeType {
    Module, Task, Function, Begin, Fork,
}

/// vitamin VCD 생성기 — 시뮬레이션 전역 단일 인스턴스
pub struct VcdWriter<W: Write> {
    writer: W,
    next_id: IdCode,
    /// signal_id → IdCode 매핑 (signal_id는 시뮬 엔진 내부 식별자)
    id_map: HashMap<u64, IdCode>,
    current_time: u64,
    active: bool,        // $dumpoff 상태
    enabled: bool,       // $dumpfile + $dumpvars 호출 완료 여부
    byte_limit: Option<u64>,
    bytes_written: u64,
}

impl<W: Write> VcdWriter<W> {
    pub fn new(writer: W) -> Self { ... }

    // ── 헤더 단계 ──────────────────────────────────────────

    /// $date 블록 기록
    pub fn write_date(&mut self, date: &str) -> std::io::Result<()>;

    /// $version 블록 기록
    pub fn write_version(&mut self, version: &str) -> std::io::Result<()>;

    /// $timescale 블록 기록. unit 예: "1ns", "10ps"
    pub fn write_timescale(&mut self, unit: &str) -> std::io::Result<()>;

    /// $scope 열기
    pub fn begin_scope(&mut self, scope_type: ScopeType, name: &str) -> std::io::Result<()>;

    /// $upscope
    pub fn end_scope(&mut self) -> std::io::Result<()>;

    /// $var 선언. signal_id에 IdCode를 할당하고 반환
    pub fn declare_var(
        &mut self,
        var_type: VarType,
        width: u32,
        reference: &str,
        signal_id: u64,
    ) -> std::io::Result<IdCode>;

    /// $enddefinitions $end
    pub fn end_definitions(&mut self) -> std::io::Result<()>;

    // ── 초기 덤프 ──────────────────────────────────────────

    /// $dumpvars 블록 시작
    pub fn begin_initial_dump(&mut self) -> std::io::Result<()>;

    /// $dumpvars 블록 종료 ($end)
    pub fn end_initial_dump(&mut self) -> std::io::Result<()>;

    // ── 시뮬레이션 데이터 ─────────────────────────────────

    /// #<time> 타임스탬프 기록. time은 단조 증가해야 함
    pub fn timestamp(&mut self, time: u64) -> std::io::Result<()>;

    /// 스칼라 값 변화: `<0|1|x|z><id>` (공백 없음)
    pub fn change_scalar(&mut self, signal_id: u64, value: ScalarValue) -> std::io::Result<()>;

    /// 벡터 값 변화: `b<bits> <id>`
    pub fn change_vector(&mut self, signal_id: u64, bits: &[Bit]) -> std::io::Result<()>;

    /// 실수 값 변화: `r<%.16g> <id>`
    pub fn change_real(&mut self, signal_id: u64, value: f64) -> std::io::Result<()>;

    // ── dump 제어 ─────────────────────────────────────────

    /// $dumpall: 모든 추적 변수 현재값 강제 기록
    pub fn dump_all(&mut self, state: &SimState) -> std::io::Result<()>;

    /// $dumpoff: 모든 변수에 x/bx 기록 후 추적 중단
    pub fn dump_off(&mut self, state: &SimState) -> std::io::Result<()>;

    /// $dumpon: 덤프 재개, 현재 값 기록
    pub fn dump_on(&mut self, state: &SimState) -> std::io::Result<()>;

    /// $dumpflush: 버퍼 플러시
    pub fn flush(&mut self) -> std::io::Result<()>;

    /// $dumplimit: 파일 크기 한도(바이트) 설정
    pub fn set_limit(&mut self, bytes: u64);

    // ── 내부 유틸 ─────────────────────────────────────────

    /// $comment 블록 기록
    pub fn write_comment(&mut self, text: &str) -> std::io::Result<()>;

    /// signal_id로 IdCode 조회 (없으면 None)
    pub fn id_for(&self, signal_id: u64) -> Option<IdCode>;

    /// 바이트 한도 도달 여부 확인
    pub fn is_limit_reached(&self) -> bool;
}
```

### dump 태스크 라우팅

`hdl-builtins` 크레이트의 dump 카테고리 핸들러가 `VcdWriter` 메서드를 호출한다. `VcdWriter`는 시뮬레이션 전역 단일 인스턴스(`Arc<Mutex<Option<VcdWriter>>>`)로 관리된다.

```
$dumpfile("x.vcd") 핸들러
  → VcdWriter::new(File::create("x.vcd"))
  → 전역 인스턴스에 저장

$dumpvars(0, tb) 핸들러
  → scope 트리 순회 → declare_var() per signal
  → end_definitions()
  → begin_initial_dump() → 각 신호 초기값 기록 → end_initial_dump()

시뮬 루프에서 신호 변화 감지 시
  → VcdWriter::timestamp(current_time) (필요시)
  → change_scalar / change_vector / change_real

$dumpoff 핸들러  → VcdWriter::dump_off(state)
$dumpon 핸들러   → VcdWriter::dump_on(state)
$dumpall 핸들러  → VcdWriter::dump_all(state)
$dumpflush 핸들러 → VcdWriter::flush()
$dumplimit(N) 핸들러 → VcdWriter::set_limit(N)
```

시뮬레이터 종료 시 `VcdWriter::flush()`를 자동 호출해 미기록 버퍼를 플러시하고 최종 newline을 보장한다.

## 검증 전략

### 1단계 — GTKWave / Surfer 로드

생성한 VCD를 GTKWave와 Surfer(웹 기반 파형 뷰어)로 로드해 파싱 오류가 없음을 확인한다. 오류 없이 파형이 표시되면 형식 적합성 1차 통과.

### 2단계 — iverilog 정규화 diff

동일한 테스트벤치를 iverilog로 실행한 VCD와 vitamin이 생성한 VCD를 비교한다. 식별자 코드는 시뮬레이터마다 다를 수 있으므로 정규화 과정이 필요하다:

```
정규화 절차:
1. 각 VCD에서 $var 선언을 파싱해 signal_name → id_code 매핑 구축
2. 값 변화 레코드를 id_code 대신 signal_name 키로 변환
3. 동일 타임스탬프 내 변화 레코드를 signal_name 기준으로 정렬
4. 정규화된 두 레코드를 라인별 diff
```

식별자 코드 차이를 흡수한 후 값 변화 내용이 완전히 일치해야 한다.

### 3단계 — Verilator VCD 비교 (옵션)

Verilator도 동일한 정규화 diff로 비교한다. Phase 1에서는 iverilog 기준 일치를 우선 목표로 한다.

---

## 비목표 (현 단계)

- **확장 VCD (`$dumpports*`)**: 포트 강도(strength) 정보 포함 상용 툴 확장. vitamin Phase 1 범위 밖.
- **FST 포맷**: GTKWave 네이티브 압축 포맷. VCD 호환 완성 후 Phase 2+ 고려.
- **LXT/LXT2**: 레거시 GTKWave 포맷. 지원 계획 없음.
