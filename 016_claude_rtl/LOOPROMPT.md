## 1. 아이템 선택 (ROADMAP에서 1개)
- `docs/ROADMAP.md` 잔여 항목 확인(특히 §4.5.x의 발굴된 pre-existing silent-wrong·개발예정 목록).
- **선택 전 후보를 오라클로 재현 검증**: 문서화된 "미수정 버그"가 이미 오라클과 일치할 수 있음(misdiagnosed·이전 슬라이스가 부수 수정·특정 트리거 필요). 재현 안 되면 비목표로 기록하고 다음 후보로. (이번 루프: `@*` t0·`@*` Comb self-write·#5 mixed-edge가 전부 이미 일치였음.)
- **recorded 후보는 mechanism-level로 재그라운딩—기록된 증상이 실제 버그보다 좁을 수 있음**(이번 루프 §4.5.21): "32-bit decimal 필드폭 10 vs 11"로 기록됐으나 **param 공간(폭×signedness) sweep**이 실제 근본=**signed `%d`가 sign column 미예약**(8/16/32 전 signed 영향·32-bit specific 아님)임을 드러냄. 후보 착수 시 기록 framing을 믿지 말고 **변수 공간을 sweep**(width·signedness·value·edge)해 정확한 룰을 핀→슬라이스가 좁게 기록됐어도 fix는 일반해(localized 1-fn)로 더 큰 가치.
- **우선순위: ① 오라클 있는 CRITICAL silent-wrong > ② 오라클 있는 기능 갭(loud→supported) > ③ 전제조건 충족된 honest-loud 승격.** 오라클 없음·전제조건 미충족(예: arm-slot 추적·deferred hier-edge-sens 필요)은 **건드리지 말고 honest-loud 유지**.
- **쉬운 silent-wrong 소진 시 ②(loud→supported)로 전환**: vita가 loud-reject하나 iverilog가 지원하는 흔한 구문을 오라클로 찾을 것(예: multi-driver tristate). **loud→supported는 additive라 byte-identity가 강력**(거부되던 구문은 기존 디자인에 전무)=silent-wrong 수정보다 저위험. ⚠️ **단, elaborate가 허용을 시작하는 집합 = 엔진이 실제 처리하는 집합이 EXACTLY 동일해야 함**(불일치=허용됐으나 미처리=last-wins silent-wrong). 둘의 eligibility 술어를 같게 쓰고 적대 리뷰로 집합 동일성을 검증.
- **feature가 '거부된다'고 새 인프라 슬라이스로 잡기 전에 EXISTING 부분지원을 grep**(이번 루프 named event=최대 교훈): vita가 `@e`를 parse-reject했으나 grep이 event 인프라 **이미 완비**(`NetVarKind::Event`·`->`=`parse_trigger_stmt`·`@(e)`=`parse_event_stmt`·event-as-value 이미 loud)를 드러냄→유일 갭이 **bare 괄호생략 1개**=engine 신규 0의 1-site 파서 fix(moderate feature→tiny additive로 축소). **실패하는 EXACT sub-form만 격리**해 probe(`@e` vs `@(e)`)→어느 sub-form이 정확히 막혔는지 핀하고 기존 인프라와 대조(`grep 'fn parse_<feat>'`·관련 AST kind·관련 토큰). 갭은 'feature 전체'가 아니라 'sub-form 1개'인 경우가 흔함=슬라이스 대폭 축소.
- **'지원되는' 듯한(무에러) system task도 silent-wrong일 수 있다=empty/blank/wrong 출력을 적극 의심**(이번 루프 `$swrite`): vita 진단 비대칭(unknown `$task`=W3056 **silent-skip** vs unknown `$func`=E3009-loud) 때문에 미매핑 `$task`는 조용히 no-op→타깃 unchanged. ② probe 시 vita가 거부 안 하나 iverilog와 출력이 다른(특히 **빈 문자열·blank·default 값**) 케이스 = priority-① silent-wrong 후보(②보다 우선). `grep map_systask`로 미매핑 task 열거.
- **"byte-identity 리스크가 커 보여 defer"는 측정으로 검증**: 우려가 과대평가일 수 있음(이번 루프: narrow-posedge 와이드화의 "다수 골든 flip" 우려가 측정상 미현실화=t0 무영향·전체 스위트 무회귀). defer 전 실제 영향 케이스를 오라클로 핀.

- **코어 견고(probe가 vita-ahead/no-oracle/SAME뿐)+잔여 갭이 전부 moderate-risk(deep·byte-identity flip·murky semantics·cosmetic·전용-인프라 필요)면, 억지 code 슬라이스 대신 thorough 그라운딩-기록 + LOOPROMPT 정리 iteration도 valid**(이번 루프 #29: struct `'{}`=`StructLayout`이 parser-only라 elaborate intercept 불가·decl-init은 `pending_var_inits` 경유=statement/decl-init scope 불일치=moderate 전용 슬라이스→그라운딩만 ROADMAP 기록). rush 금지(§2 '불완전 슬라이스보다 깨끗한 defer')의 iteration-레벨 적용=불완전 code보다 record/defer 우월. ⚠️ **probe로 코어 견고+잔여 moderate를 실제 확인한 뒤에만**(쉬운 ②/① 갭 남았는데 회피는 금지)·그라운딩은 다음 iteration이 즉시 착수 가능하게 정확한 난점(어느 함수·어느 scope 불일치)까지 기록. **그라운딩-기록은 다음 iteration이 prerequisite silent-wrong을 발굴하게 만든다=defer가 낭비 아님**(루프 #30이 #29의 defer 검증): deferred moderate feature(struct `'{}`)의 desugar를 핀하다(=`s='{e0,e1}`≡`{w0'(e0),w1'(e1)}`) **그 desugar의 prerequisite인 좁은 silent-wrong**(cast `8'('1)`=01)을 발굴—이건 그 자체로 **오라클 있는 ① 슬라이스**(moderate feature보다 우선·저위험)이자 **deferred feature를 unblock**. 즉 moderate feature 직접 착수가 막히면 그 desugar를 building-block 단위로 핀→각 block(cast·concat·width)이 독립 ① 슬라이스가 됨(큰 feature=작은 prerequisite fix들의 합).

- **probe의 'DIFFER'는 4-way 분류 후에만 슬라이스 대상—오분류=슬라이스 낭비**(이번 루프 port-conn sweep): 차이를 (a)**오라클 있는 진짜 갭**[`.name`/`.*` parse-reject] (b)**no-oracle**=iverilog도 거부[streaming concat=`sorry: not supported`]→honest-loud 이미 correct·skip (c)**vita-ahead**=vita 지원·iverilog 거부[`$typename`]→silent-wrong 아님·유지(오라클 없어 검증 불가=손대지 말 것) (d)**harness-formatting-only**=vita `simulation ended (Finish)` vs iverilog `$finish called at N`·iverilog `vvp.tgt sorry: ... ignored` 경고→**정규화 후 재비교**(실제 `$display`/`$monitor` 출력만 비교)로 분류. **정규화 없이 (d)를 갭으로 오인하면 첫 batch 전체가 가짜 DIFFER**(이번 루프 urandom_range·unique-case가 정규화 후 SAME). probe harness는 처음부터 finish/sorry 라인 strip하는 `norm()` 필터 내장. (a)만 착수. **(e)iverilog가 자기 자신과 INCONSISTENT면 vita의 spec-correct가 target—iverilog 아님**(루프 #43): 같은 식이 두 position서 다른 답 주면(`'1` default-arg=iverilog 1-bit self-det vs explicit-actual=formal 폭·signed-cast default=sign drop vs explicit=sign-extend) iverilog 버그→vita가 spec+자기 일관이면 그게 correct·DIFFER는 non-target. 판별=같은 식을 explicit actual로도 줘 iverilog 답이 바뀌나 확인(바뀌면 iverilog self-inconsistent).
- **갭이 관련 기능 cluster면 work가 어디 떨어지는지(risk-tier)로 분할**(이번 루프 port-conn=`.name`+`.*`): 같은 cluster(포트연결 편의구문)라도 `.name`=**파서-only desugar**(`.port`≡`.port(port)`=clean·additive·elaborate 무변경) vs `.*`=**elaborate-time 포트 introspection 필요**(미연결 포트 전수를 same-name 자동연결=callee 포트리스트 알아야 함)=risk-tier 상이→**저위험(파서) 먼저 착수·고위험(elaborate) 별개 슬라이스 기록**. cluster를 한 슬라이스에 욱여넣지 말 것(format cluster는 동일 fmt 함수라 묶였지만, 여기처럼 파이프라인 단계가 다르면 분할이 옳음).
- 선택 즉시 격리 브랜치: `git checkout -b feat-<slug>` (main에서 직접 구현 금지). **첫 EDIT 전에 브랜치 생성=절차 1순위**(루프 #48 슬립: 그라운딩 몰입하다 브랜치 없이 main서 편집→커밋 전 발견·`git checkout -b`가 uncommitted 변경 carry해 무손실 복구했으나, 커밋까지 갔으면 strand 위험). 그라운딩(probe)은 read-only라 브랜치 전에 해도 무방하나 **소스 첫 수정 직전 `git rev-parse --abbrev-ref HEAD`로 main 아님을 확인**.

- **murky해 보이는 후보는 stimulus 경로를 바꿔 재그라운딩—murkiness가 feature가 아닌 probe artifact일 수 있음**(이번 루프 §4.5.22 %0s): 직전 반복서 embedded-null이 깨져 보였으나(`[A Bc]` vs `[  Bc]`) 그건 **part-select 대입**(`s[8*4:8*3+1]="A"`)의 별개 버그였고, **whole-assign**(`s=32'h41004263`)으로 재그라운딩하니 embedded-null MATCH=feature는 clean(leading-null strip만). 같은 값을 다른 경로(whole-assign vs part-select·리터럴 vs 변수·직접 vs 함수반환)로 만들어 차이가 feature냐 stimulus냐 격리.
- **multi-construct repro가 틀리면 각 construct/path를 독립 테스트로 BISECT해 정확한 failing 조합을 핀**(루프 #35 array-init): `int a[4]='{1,2,3,4}; foreach() sum`=0이 array-decl-init 버그냐 foreach 버그냐를 **각각 독립 probe**(foreach-only=정상·array-decl-init-only=0)로 격리→array-decl-init 확정. 이어 **축별 bisect**: scope(module-scope=정상 vs block-local=버그)·path(statement `a='{…}`=정상 vs decl-init=버그)→정확한 failing 조합=**block-local + decl-init + unpacked-array**. 한 축씩 변화시켜 working/broken 경계를 좁히면 fix 지점이 명확(여기선 `hoist_block_local_nets` vs `collect_var_init_drivers` 비대칭). 여러 construct가 얽힌 repro를 통째로 디버그하지 말 것=어느 construct가 범인인지 먼저 bisect.
- 관련 코드(파이프라인 해당 단계)를 정독 → **byte-identity 논증**(이 변경이 비대상 디자인서 왜 무영향인지)을 먼저 세움. 못 세우면 범위를 좁히거나 honest-loud.
- **inline-path silent-wrong은 frame-path(`automatic` 버전)를 오라클-보조 정답 레퍼런스로**(루프 #38 inline-func-ret-width): 같은 함수의 `function automatic …` 버전을 vita에 돌리면 frame/net path가 engine resize로 CORRECT 동작(inline은 순수 ExprId·net write 無라 §10.7 resize 누락)→정답 핀(+잠재 fix 전략=route-to-frame). 단 routing은 `@*` sensitivity·VCD·timing 변경=value-resize 슬라이스엔 고위험→**byte-identity 예측 가능한 surgical fix 우선**. **width/sign-resize fix의 byte-identity는 `ir_bits_of(natural rhs) vs 선언 폭`으로 사전 예측**(no-op 조건=rw==w && sign 일치): `a+b`(8-bit)=`max(8,8)`=8=선언 8→무변경(common case)·`f=42`(32-bit literal)→선언 8=truncate=변경(=fix). 구현 전 `ir_bits_of` 룰(Add/Mul=max-operand·shift=left-operand만·literal=32)로 어느 케이스가 no-op인지 핀→high-blast-radius 변경 de-risk. **resize는 RHS 자기 sign으로 extend·TARGET sign으로 stamp**(루프 #38 핀: signed local widen=sign-extend·unsigned→signed wide=zero-extend후 $signed; `lower_prim_cast`가 모델).
- **format/rendering 서브시스템서 silent-wrong 1개 발견 시 클러스터를 의심—전 spec×flag 매트릭스를 sweep**(이번 루프 string-format 3연속: signed-fieldwidth→`%0s`→`%Ns`): 같은 fmt 코드가 `%d/%s/%h`×width×0-flag×signedness×x/z를 처리하므로 한 버그는 인접 버그의 신호. 매트릭스(spec letter × field-width{none/0/N} × 타입{signed/unsigned/packed/string/literal} × value{0/min/max/x/z/overflow})를 오라클로 전수 차분→cluster를 한 번에 발굴. **연속 슬라이스는 직전 그라운딩이 다음 오라클을 푼다**(직전 루프 `%05s`=오라클 없어 base 유지했으나 이번 `%Ns` 그라운딩이 `%05s`=strip+pad-to-5 오라클 제공). rendered byte-string의 column 폭은 `chars().count()`(byte당 1 char) 사용—`len()`(UTF-8 byte)은 ≥0x80 byte서 overcount. **단 family를 batch-handle 시 멤버가 uniform하다 가정 금지—각 멤버를 개별 오라클 핀**(이번 루프 uppercase: `%X`≡`%x`·`%S`/`%C`/`%M`=별칭·`%E`/`%G`=exponent 대문자·**`%F`=비대문자**[inf/nan 소문자 유지]). "uppercase=전부 대문자화" 같은 일괄 규칙은 differential이 깸(`%F` over-uppercase 발굴)→spec letter마다 오라클 동작 확인 후 가드 좁힘. **cluster 소진 신호=in-cluster 마지막 갭+인접 영역 probe가 모두 clean/done**(이번 루프 format 5연속[signed-fieldwidth→`%0s`→`%Ns`→uppercase→`%v`] 완결·non-format 6 probe 전부 SAME)→다음 반복은 **새 영역으로 branch-out 또는 deferred deep 후보**(§4.5.18 frame inner-block collision=14탄부터 deferred). 한 cluster에 무한 머물지 말 것. **cluster를 여러 iteration에 걸쳐 완결하는 것도 valid(각 piece가 oracle-backed+자연 완결일 때)이나 완결 직후 MUST branch-out**(루프 #30~33 struct `'{}`=4-iteration 완결[cast-fill prereq→scalar→cont/force/for→array-elem]·**cluster DONE→다음 iteration은 struct 회피, 새 영역 probe 필수**). **clean 슬라이스의 differential 광범위 probe도 인접 pre-existing 버그 수확**(루프 #33 task comma-typed port `task t(input logic[7:0] x,y)`가 `y`=1-bit silent-wrong[function port는 정상]=별개 ① 후보 ROADMAP 기록·fold 금지=§6).
- **desugar 슬라이스는 인프라 구축 전에 desugar EXPRESSION 자체를 sim서 검증**(이번 루프 enum `.name()`): 소스 구문뿐 아니라 desugar가 생성할 표현식(예: string/value ternary chain)을 vita에 직접 돌려 결과 정확성 확인. building block이 '돈다'고 의미가 맞는 건 아님(packed-string ternary는 짧은 라벨을 패딩→iverilog dynamic string과 불일치)→불일치 메서드/형태는 **honest-loud로 격리**(메서드-패밀리는 per-method 오라클 지원 기준 분할; 5/6 정확+1 loud ≫ 6/6 중 1 silent-wrong).
- **context-determined 기능(의미가 타깃 타입에 의존)은 ALL 변종을 구현 전 오라클로 전수 핀**(이번 루프 assignment pattern `'{…}` 중단·revert): "packed='{…}≡concat" 같은 **예측을 측정 없이 구현에 착수→라이브 차분서 오류**(`'{1,0,1,1}` on `logic[3:0]`=각 1비트 `1011`, concat-of-int=`0001`). target packed-width sizing(`W/N`)·unpacked array 방향·비-assign-context loud·미파싱 변종(repl/default/named) 전부 **사전 핀**. 큰 의미공간이면 **전용 슬라이스로 분리·이번 반복 비목표**로 기록(rush 금지=불완전 슬라이스보다 깨끗한 defer가 우월).
- 계획을 scratchpad에 durable 기록(파일·메커니즘·테스트 매트릭스·loud 유지선).

## 3. 구현
- 가능한 IR-0. 단일 write/엣지 등 **공통경로엔 청크포인트가 하나인지 먼저 확인**(인터프리터+VM 공유 funnel 여부). 모든 동치 경로를 빠짐없이 커버.
- 비-대상 디자인은 byte-identical 유지(가드/사이드카=값 다를 때만).
- **순수 파서 desugar(기존 AST 노드 재사용)는 최저위험 loud→supported**(inc/compound·break/continue): 신규 구문이 기존 구문의 shorthand면(`i+=e`≡`i=i+e`·`i++`≡`i=i+1`) elaborate/IR 손대지 말고 **파서서 기존 `Stmt::Blocking` 등으로 desugar**→AST/sim-ir/`.vu` 해시 전부 불변(신규 AST 타입 0). lvalue가 양변에 등장하면 `lvalue_to_expr`(역 `expr_to_lvalue`)로 rhs 합성·precedence는 operand를 `expr(0)` 전체 파스 후 wrap. 부수효과 expr 형태(`a=i++`)는 Pratt expr 파서에 미추가→loud 유지(scope 분리).
- **feature를 1-instance→N-instance로 확장 = 기존 per-unit lowering을 DISCRIMINATOR-arg로 파라미터화 + DISCRIMINATOR로 BRANCH(restructure 금지=byte-identity)**(루프 #40 multi-dim foreach): single-index `foreach(a[i])`=`arr.first/next(idx)`(engine iterator over dim 1)였음→multi-dim은 **dimension arg 추가**(`arr.first/next(idx, DIM)`)+elaborate handler를 dim-aware로(`net_dim_extents.get(dim-1)`). **핵심 byte-identity 기법**: 기존 경로를 재구성하지 말고 **discriminator(여기선 comma)에서 BRANCH**—comma 없으면 기존 single-index 코드 verbatim(1-arg=dim 1=`.get(0)`≡`.first()`=증명가능 동치)·comma면 신규 multi-dim 경로. soundness가 "comma 없을 때 기존 AST 동일" 전수증명. N-instance가 nesting이면(multi-dim=nested loop) **leftmost=outermost**·각 레벨 synth name은 variant간 distinct prefix(single=`__foreach_st_<L>`·multi=`__foreach_st<N>_<L>`). 신규 lowering이 non-기본 instance(dynamic array·dim 초과)에 도달하면 **honest-loud**(fixed-array만 per-dim bounds).
- **1-D→N-D 확장 전에 기존 OFFSET/INDEX 테이블이 이미 N-D를 처리하는지 확인—그러면 확장=입력을 그 테이블에 맞게 flatten뿐 + DIRECTION은 정확히 1곳서만 적용**(루프 #42 multi-dim array pattern): 1-D array `'{…}` 패턴이 multi-dim서 loud였으나 **`residual_word_offsets`가 이미 row-major multi-dim flat offset 생성**(per-dim asc/desc 존중)→확장=신규 `flatten_assign_pattern`(중첩 패턴을 declaration-order leaf로 flatten·각 레벨 count 검증)뿐, offset 계산 재사용. **핵심 silent-wrong 회피=DIRECTION 1곳 적용**: flatten은 **direction-AGNOSTIC**(순수 declaration order)·offset table만 desc 적용→leaf[k]↔offset[k] 정렬. flatten이 desc 또 적용하면 **double-flip silent-wrong**(descending서 값 오배치). 확장 전 `grep`로 기존 stride/offset/coord 헬퍼가 multi-dim 인자를 받는지 확인—받으면 코어 재구현 0.
- **control-flow 구문(loop/break/continue/early-exit)은 'intuitive' 의미 가정 금지—오라클로 break/continue 타깃을 핀**(루프 #40 multi-dim foreach=silent-wrong 회피): `foreach(a[i,j]) break`이 "전체 foreach exit" 같지만 **iverilog nested-for desugar는 break=INNERMOST 루프만 exit**(`a[3][3]` break@(1,1)=7 not 4)·continue=innermost advance. 가정대로 "전체 exit"로 구현했으면 silent-wrong. desugar 슬라이스서 break/continue/return/disable이 어느 스코프를 타깃하는지 **반드시 오라클 측정**(nested 케이스로)·nested면 innermost에만 `wrap_break` 적용·outer는 plain.
- **AST 노드 추가 전에 "기존 end-to-end 메커니즘이 이미 그 일을 하는지" 확인**(이번 루프 break/continue): control-flow도 기존 구문으로 표현되면 재사용=elaborate/engine 신규 경로 0. 예: `break`/`continue`는 `disable <label>`→Goto(파서+elaborate+engine 完備)로 desugar→IR/엔진 무변경(AST 노드+loop-target-stack 설계보다 우위). 사전에 `grep`로 등가 메커니즘(disable·goto·기존 desugar) 존재 확인.
- **미매핑 system task는 새 핸들러 작성 전에 의미상 근접한 기존 `SysTaskId` 핸들러로 라우팅 가능한지 확인**(이번 루프 `$swrite`→`$sformat`): 공유 포맷/eval 헬퍼(`format_args_str` 등)가 이미 원하는 동작($write-style append 등)을 하면 `$swrite`≡`$sformat`=**2-line 매핑**(이름→기존 SysTaskId+radix). 엔진 핸들러+공유 헬퍼를 **먼저 정독**해 '이미 그 일을 하는지' 확인 후 매핑(= §3 'AST 노드 추가 전 기존 메커니즘 확인'의 system-task판). arg 레이아웃(dest/fmt/radix 추출)이 기존 task와 호환되는지 전 변종 핀.
- **같은 구문의 N개 parser 변종은 ALL을 공유 헬퍼로 라우팅**(do-while 누락=silent-wrong·루프 #32 assign-RHS=완결 적용): 루프처럼 변종이 여러 파서 함수면(for/while/repeat/forever/foreach + **do-while**=별도 `parse_do_while`) `grep 'fn parse_'`로 전수 열거→하나라도 공유 헬퍼(`parse_loop_body`) 미경유 시 그 변종서 desugar 미적용=잘못된 타깃/드롭(do-while 안 break가 enclosing 루프 종료=silent-wrong). = §3 'ALL elaborate 컨텍스트 전수'의 **parser-변종판**. (do-while: soundness은 'honest-loud'로 분류했으나 differential이 중첩 시 silent-wrong 입증→differential 우선.) **'assign-RHS'는 ≥7-site construct-class**(루프 #32 struct `'{}`): blocking `=`·nonblocking `<=`·decl-init·continuous `assign`·procedural-continuous `assign`·`force`·for-init/step—파서 헬퍼(`maybe_*_rhs`)를 ALL site에 적용해야 일관(빠진 site=그 컨텍스트만 loud gap=혼란). **완결성 체크 = soundness 서브에이전트에게 'construct-class의 EVERY site 열거+각 hook 有無/필요 판정' 명시 의뢰**(루프 #32: soundness가 `parse_for_assign` missing 발굴=differential은 못 잡음[loud는 silent-wrong 아님→probe 무관심]·soundness의 site-enumeration이 MISSING-site 전담). **reviewer가 찾은 loud-gap이 동일 validated 메커니즘으로 닫히면(1-line·오라클 검증·byte-identity 확인) 같은 슬라이스서 fold**(defer 말 것=부분 feature 혼란 방지). = §4 differential(EXISTING path silent-wrong)↔soundness(MISSING path/site)의 또 다른 COMPLEMENTARY 축.
- **loud→supported 파서 fix는 오라클로 acceptance boundary를 전수 핀한 뒤 parse primitive를 고름**(이번 루프 bare `@e`): iverilog가 `@e`/`@clk`/`@u.s`/`@a[2]`=accept·`@posedge clk`(bare edge)/`@a+b`/`@a&&b`=reject임을 핀→이게 `expr_postfix()`(primary+postfix 참조, binary infix서 멈춤)를 고르게 함(full `expr(0)`이면 `@a+b`를 `@(a+b)`로 silently over-accept=iverilog 거부분 수용). **어느 primitive가 정확히 그 경계서 멈추는지 grep으로 확인**(`fn expr_postfix`/`fn expr_primary`/binding-power 표)한 뒤 선택. keyword guard도 경계의 일부(`posedge`=키워드→`is_ident()` false→bare edge 자동 loud).
- **identifier-led 모호 opener(decl `T name` vs statement `name …`)는 SECOND-token lookahead로 disambiguate—`<ident> <ident>`는 모호성 0 decl**(루프 #36 block-local typedef decl=② parse gap): user typedef명으로 시작하는 절차 블록 선언(`e x;`·`s_t s;`)이 E2002였음—decl 루프가 built-in 키워드만 인식. 핵심 disambiguation: `peek_at(1)`이 ident면 decl(어떤 statement도 `ident ident` 형태 없음)·`::`/`'`/`(`/`[`/`.`/`=`/`<=`/`++`면 statement→fall-through. **기존 recognizer를 미러하되**(module-item 파서의 `peek_typedef_name`→`parse_typed_decl`) **statement도 있는 컨텍스트(block/tf-body)엔 offset-1 guard 추가**(module-item엔 statement 無라 guard 불필요). **`self.typedefs`엔 타입명만**(enum-label·var는 별개 맵)=var/label 오인 불가. **검증 분담**: differential에게 **기존 RTL 모듈 byte-identity sweep**(statement 오인-grab=valid→parse-error regression 잡기)·soundness에게 **statement-opener의 offset-1 token 표 전수**(necessary&sufficient 증명)+**body-decl site 완전성**(begin/fork→block_body·task/func/class-method→tf_body, §3 line 45 ALL 변종). 순수 파서·AST 불변·IR-0·additive.
- **렉서 토큰 추가 시 logos longest-match를 prefix 공유 ALL 기존 토큰과 대조**(이번 루프 14 토큰): 다char 연산자 충돌 전수(SVA `|=>`/`|->` vs 신규 `|=`·`<<=` vs `<=`/`<<`/`<<<`·`>>>=` vs `>>>`). **필요한 모든 길이가 정의돼야** longest-match가 기존 스트림 보존(공백 없는 `a++b`만 신규 융합=iverilog도 loud=무해).
