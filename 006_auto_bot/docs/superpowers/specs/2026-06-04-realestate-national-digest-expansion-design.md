# 부동산봇 전국 권역 디제스트 확장 — 설계

**작성일:** 2026-06-04
**상태:** 설계 승인 대기 → 구현 계획(writing-plans) 예정
**전제:** v1(서울 25구)은 구현·발행 완료. 백필로 119시군구 × 4종(아파트·오피스텔 × 매매·전월세) 392만행 적재 완료. 이 스펙은 **발행 범위를 수집 범위(`ALL_REGIONS`)에 맞추는 v2**.

## 1. 목표

주간 디제스트를 **서울 전용 → 전국 119시군구 단일 블로그 글**로 확장한다. 서울은 현행 상세를 유지하고, 비서울은 권역 요약으로 덧붙여 이미 수집해 둔 경기·광역시·세종 데이터를 활용한다. 발행 주기·경로(토 01:00, 직접 MCP, Blogger+Telegram)는 불변.

## 2. 확정된 범위 결정

| 항목 | 결정 |
|------|------|
| 글 구조 | 전국 단일 글, 권역 섹션 |
| 권역 상세도 | 하이브리드 — 서울 전구 상세 + 비서울 권역 요약 |
| 비서울 오피스텔 | **경기·광역시 건수** (세종만 제외) |
| 시황 해석 | **전국 본격 다문단** (서울·경기·광역시 고르게) |
| 블로그 제목 | `날짜, N월 M주차 {AI 헤드라인}` (주차별 일관) |
| 라벨 | 7~9개, 내용 반영 동적 |

## 3. 권역 그룹 (지역코드 prefix로 도출)

| prefix | 권역 | 시군구 수 |
|--------|------|-----------|
| 11 | 서울 | 25 |
| 41 | 경기 | 44 |
| 26 / 27 / 28 / 29 / 30 / 31 | 부산 / 대구 / 인천 / 광주 / 대전 / 울산 | 50 |
| 36 | 세종 | 1 |

`region_code[:2]`로 권역 판정. 광역시는 prefix→시명 매핑으로 시별 소그룹.

## 4. 블로그 글 구조 (한 편, 깔때기)

1. **전국 헤더 요약** — 총 신규·신고가 비중(%) + 권역별 한 줄(서울/경기/광역시/세종)
2. **서울 (상세)** — 현행 그대로: 25구 온도차 표 + 신고가/신저점 단지 하이라이트 + 전세가율 + 오피스텔(매매+전월세)
3. **경기 (요약)** — 경기 44 합산 신규·신고가비중·평균 전세가율·오피스텔 건수 + 뜨거운 시군구 top 5 + 신고가/신저점 하이라이트 top 3
4. **6대 광역시 (요약)** — 시별(부산·대구·인천·광주·대전·울산) 한 줄 요약(신규·신고가비중·전세가율·오피스텔 건수) + 각 시 top 구 + 하이라이트
5. **세종 (요약)** — 한 줄(신규·신고가비중·전세가율)
6. **시황 해석 (Gemini, 전국 다문단)**
7. **caveat** (국토부 실거래·최근월 미확정·믹스보정 기준)

## 5. 데이터 깊이

| 권역 | 아파트 매매 | 전세가율 | 오피스텔 |
|------|------------|---------|---------|
| 서울 | 온도차·신고가·신저점·믹스보정·하이라이트 | ✓ | 매매+전월세 상세 |
| 경기 | 신규·신고가비중·top 이동 시·하이라이트 | ✓ | 매매+전월세 건수 |
| 광역시 | 신규·신고가비중·top 구·하이라이트 | ✓ | 매매+전월세 건수 |
| 세종 | 신규·신고가비중 | ✓ | ✗ |

## 6. 구현 (모듈 경계)

- **`realestate_bot/regions_extra.py`** — `REGION_GROUPS`(prefix→권역명) + `group_of(region_code)` 헬퍼 + 광역시 prefix→시명 맵.
- **`realestate_bot/indicators.py`** — `rollup_groups(per_gu, jeonse, officetel, group_fn)` 신규 순수함수. 권역별 `{new_total, high_total, high_pct, avg_jeonse, top_movers[(gu,stats)], officetel_count}` 반환. `top_movers`는 기존 `rank_regions`와 동일 기준(신고가 비중 → 신규 건수 내림차순) 재사용.
- **`weekly_realestate_bot.py`**
  - `run()`: `SEOUL_GU` → `ALL_REGIONS`로 `build_report`·`_collect_extra`·`synthesize` 호출.
  - `_collect_extra`: 전세가율용 아파트 전월세는 **전 지역**, 오피스텔(매매+전월세)은 **서울·경기·광역시**(세종 제외) 적재.
  - `_convert_html`: Claude 반환 `(html, blog_title)`에서 **blog_title(AI 헤드라인) 보존**(현재 `_`로 버림) → 제목 조립에 사용.
  - commentary 입력에 권역 rollup 추가. 제목·라벨 동적 산출(§11).
- **`realestate_bot/digest.py`** — 권역 요약 렌더 함수 신규. 서울 상세는 기존 로직 재사용. 데이터 없는 권역은 degrade(섹션 생략).
- **`realestate_bot/commentary.py`** — 프롬프트를 전국 다문단(서울·경기·광역시 고르게)으로, 입력 schema에 권역 rollup 반영. 실패 시 빈 문자열 degrade(불변).
- **하이브리드 원칙 불변**: 숫자는 전부 코드, 해석만 Gemini, HTML 변환만 Claude.

## 7. 데이터 흐름

`run()` → `MCPClient`(직접, Claude 한도 0):
1. `build_report(ALL_REGIONS, [최신월, 직전월])` → `per_gu`(119) + 전국 highlights
2. `_collect_extra`(아파트 전월세 전 지역 + 오피스텔 서울·경기·광역시) 적재
3. `synthesize(ALL_REGIONS, 직전월)` → 구별 전세가율·오피스텔 건수
4. `rollup_groups` → 권역 집계 + top movers
5. `digest.build_digest` → 전국 markdown
6. `commentary`(전국 rollup) → 시황 다문단
7. Claude HTML 변환 → Blogger 업로드 + Telegram 알림

## 8. 성능

- 주간 fetch ≈ 매매 238(119×2월) + 아파트 전월세 238 + 오피스텔(서울25+경기44+광역시50)×2월×2종 ≈ 476 → 총 **~950 직접 MCP 호출(~4–6분)**.
- Claude HTML: 글이 길어져 청크 수↑ → 변환 ~7–12분 예상. 토 01:00 1회라 허용.
- MOLIT 일일 호출 한도: 주 1회·2개월 범위라 안전(백필과 달리 36개월 아님).
- fail-safe: 권역/데이터 종류별 fetch 실패는 degrade(해당 섹션·지표 생략), 본 디제스트는 계속 진행.

## 9. 테스트

- `group_of` / `REGION_GROUPS` 매핑(11/41/26~31/36).
- `rollup_groups` 단위: 권역 집계, top movers 정렬, 데이터 없는 권역 degrade.
- `digest` 권역 섹션 렌더: 데이터 유무 degrade, 서울 상세 불변 회귀.
- `synthesize` 전국(비서울 포함) 집계.
- 기존 56 테스트 통과 유지 + 신규 ~6–8.

## 10. 비고 / 리스크

- **첫 전국 발행도 0신규 가능**(데이터 선점) — 최초 검증은 직전과 동일하게 "최근 완료월 델타 리셋" 방식 재사용.
- 글 길이 증가 → Claude HTML 변환 시간 모니터링 필요(타임아웃 여유 확인).
- 경기 44개는 시 단위(구 분리 시는 상위 시로 롤업하지 않고 구 단위 그대로 집계) — v2에선 단순 합산, 시-단위 재롤업은 향후 과제.

## 11. 블로그 제목·라벨

### 제목 형식 (주차별 일관)

`{YYYY-MM-DD}, {N월 M주차} {AI 헤드라인}`

- 예: `2026-06-06, 6월 1주차 전국 신고가 21%, 수도권 과열·부산 냉각`
- **주차**: 발행일(토) 기준 월내 주차 = `((day-1)//7)+1` → "6월 1주차". 결정적·라이브러리 불필요.
- **AI 헤드라인**: `convert_md_to_html_via_claude` 반환 `blog_title`(현재 `_`로 버려짐) 사용. `blogger-html/SKILL.md`의 BLOG_TITLE 가이드에 헤드라인 규칙 명시 — 구체 숫자 + 가장 뜨거운/차가운 권역 대비, ≤40자, 한자·추상 단계 헤더 금지([[feedback-no-hanja-narrative-headers]]). 비면 fallback "전국 아파트 시장 흐름".
- 첫 청크 변환에서 헤드라인 확보, 나머지 청크는 본문만.

### 라벨 (7~9개, 내용 반영 동적)

- **고정(4)**: 부동산, 아파트, 실거래가, 주간시황
- **범위(2)**: 전국 + 가장 활발한 권역명(서울/경기/부산…)
- **지표(1~3)**: 전세가율 + (신고가 비중 높으면 "신고가") + (오피스텔 활발하면 "오피스텔")
- **핫스팟(1)**: 가장 뜨거운 구(예: 영등포구)
- → 총 **7~9개**. 코드가 그 주 데이터에서 동적 산출(빈 슬롯은 다음 우선순위로 채워 7개 하한 보장).
